# План отладки: изображения + бесконечный рефреш

## Диагноз проекта

**Стек:** React + Vite + `vite-plugin-pwa` (Workbox) + `registerType: "autoUpdate"`  
**Деплой:** Docker (nginx:1.25-alpine) → домен `ekbmassage.online`  
**SW-конфиг:** `frontend/vite.config.ts`  
**nginx-конфиг:** inline в `frontend/Dockerfile:10-52`  
**API-клиент:** `frontend/src/api/client.ts` (axios с 401-интерцептором)  
**Auth-флоу:** email/code (безпарольный), роуты `/auth` и `/auth/verify`

## Корневая диагностика

### Проблема 1: Изображения не загружаются — это НЕ SW, это сервер/инфраструктура

**`ERR_CONTENT_LENGTH_MISMATCH`** на статическом файле `/illustrations/landing_hero.png` означает: сервер начал отправлять ответ (200 OK), но соединение оборвалось до передачи полного тела. nginx послал `Content-Length: X`, но реально передал меньше байт.

**`ERR_CONNECTION_CLOSED`** на `fonts.gstatic.com` (внешний ресурс!) подтверждает: проблема на уровне **транспортного слоя**, а не приложения.

**`Cache.put() encountered a network error`** в sw.js — это СЛЕДСТВИЕ обрывов соединений: SW пытается закэшировать ресурс при установке, но fetch обрывается → `Cache.put()` падает.

**Три уровня возможной причины (от наиболее к наименее вероятному):**

| # | Причина | Признаки совпадают? |
|---|---|---|
| **A** | **Прокси/CDN перед Docker** (Cloudflare, хостинг-провайдер) обрывает соединения — лимит body size, таймаут, buffering | Локальные + внешние ресурсы обрываются |
| **B** | **nginx gzip сжатает уже сжатые файлы** (.png, .woff2) → Content-Length не совпадает с фактическим размером тела | Только статика с `Cache-Control` хедером |
| **C** | **Docker контейнер** нехватка памяти / лимит соединений | Прерывистые ошибки |

**Ключевой аргумент:** внешние ресурсы (Google Fonts) тоже падают с `ERR_CONNECTION_CLOSED`. Это исключает nginx как единственную причину — проблема либо у клиента (ISP, прокси), либо у CDN/прокси перед Docker.

### Проблема 2: Бесконечный рефреш на странице авторизации — баг в axios-интерцепторе

**Корень — race condition + устаревшие проверки путей:**

`frontend/src/api/client.ts:16-29`:
```js
const onAuthPage = window.location.pathname === "/login" || window.location.pathname === "/register";
const publicPages = ["/", "/login", "/register"];
const onPublicPage = publicPages.includes(window.location.pathname);
```

**Флоу бесконечного цикла:**
1. Пользователь на `/auth` → `checkAuth()` вызывает `GET /user/me` → 401
2. Интерцептор пробует `POST /auth/refresh` → fails (нет сессии)
3. `onAuthPage` = `false` (pathname `/auth`, а не `/login`)
4. `onPublicPage` = `false` (pathname `/auth` не в массиве)
5. `window.location.href = "/login"` → полный reload
6. React Router: `/login` → `<Navigate to="/auth" replace />` → URL = `/auth`
7. К моменту завершения async `checkAuth()` pathname уже `/auth` → шаг 1 повторяется

**Дополнительный фактор:** `registerType: "autoUpdate"` в SW может вызывать автоматический reload при обновлении, что усугубляет цикл.

**Примечание:** на главной странице (`/`) цикла нет, потому что `/` входит в `publicPages`. Это подтверждается логами: 401 на `/api/user/me` есть, но редиректа нет.

## Пошаговый план отладки

### Шаг 1. [КРИТИЧЕСКИЙ] Диагностика ERR_CONTENT_LENGTH_MISMATCH — серверная причина

Ошибка `ERR_CONTENT_LENGTH_MISMATCH` означает обрыв соединения при передаче тела ответа. Это НЕ баг SW — это инфраструктурная проблема.

**Проверка 1A — curl напрямую в обход браузера:**
```bash
# Сравнить Content-Length хедера с реальным размером тела
curl -v -o /dev/null https://ekbmassage.online/illustrations/landing_hero.png 2>&1 | grep -i "content-length"
curl -s https://ekbmassage.online/illustrations/landing_hero.png -o /tmp/hero.png
ls -la /tmp/hero.png
# Если размер файла НЕ совпадает с Content-Length → сервер обрывает передачу
```

**Проверка 1B — curl с сервера внутри Docker:**
```bash
# Зайти в контейнер nginx
docker exec -it <frontend-container> sh
curl -I http://localhost/illustrations/landing_hero.png
# Если Content-Length корректный внутри → проблема в прокси перед Docker
```

**Проверка 1C — проверить gzip конфликты:**
nginx (в Dockerfile) не имеет явного `gzip off`. По умолчанию `gzip on` в nginx:1.25-alpine. Проблема: nginx может gzip-сжать уже сжатые файлы (.png, .woff2), что ломает Content-Length.

```bash
# Сравнить ответы с gzip и без
curl -H "Accept-Encoding: gzip" -v https://ekbmassage.online/illustrations/landing_hero.png 2>&1 | head -30
curl -H "Accept-Encoding: identity" -v https://ekbmassage.online/illustrations/landing_hero.png 2>&1 | head -30
# Если с gzip ответ обрывается, а без gzip — нет → проблема в gzip
```

**Проверка 1D — есть ли CDN/Cloudflare перед Docker:**
```bash
nslookup ekbmassage.online
dig ekbmassage.online
# Если CNAME указывает на Cloudflare/CDN → они могут обрывать соединения
curl -v https://ekbmassage.online 2>&1 | grep -i "server\|via\|x-powered"
```

**Проверка 1E — Docker логи на предмет ошибок:**
```bash
docker logs <frontend-container> --tail 100
docker stats <frontend-container>  # проверить memory/CPU usage
```

### Шаг 4. Проверка состояния Service Worker

**Инструменты:** `DevTools → Application → Service Workers`

1. В DevTools на вкладке `Application → Service Workers`:
   - **Status**: должен быть `activated and is running`
   - Если `waiting to activate` → кликнуть `skipWaiting`
   - Если `redundant` → SW не установлен (упал при установке из-за Cache.put() ошибки)
2. Проверить `Update on reload` — если включено, SW переустанавливается при каждом reload
3. Включить `Offline` — проверить, как ведёт себя страница без сети

**Что искать:**
- SW в состоянии `redundant` → подтверждение, что `Cache.put()` падает при установке
- `navigator.serviceWorker.controller === null` → SW не контролирует страницу
- Ошибки в `Console` с тегом `sw.js`

### Шаг 5. Анализ кэша SW

**Инструменты:** `DevTools → Application → Cache Storage`

1. Открыть `Cache Storage` → должен быть `workbox-precache-v2-*`
2. Просмотреть список закэшированных URL
3. Если кэш пуст или SW redundant — это СЛЕДСТВИЕ `Cache.put() network error`

### Шаг 6. Анализ nginx-конфигурации (Dockerfile:10-52)

Текущий nginx конфиг имеет потенциальные проблемы:

```nginx
# Эта локация матчит .png файлы и добавляет Cache-Control
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
}
```

**Проблемы:**
1. Нет `gzip off` для уже сжатых форматов (.png, .woff2, .jpg)
2. Нет `proxy_pass` — файлы отдаются как статика, что правильно
3. Нет явной настройки `sendfile`, `tcp_nopush`
4. Нет `worker_connections` и `worker_rlimit_nofile` лимитов

(async () => {

})();

## Типичные паттерны, вызывающие ERR_CONTENT_LENGTH_MISMATCH

### 1. nginx gzip сжимает уже сжатые файлы
nginx default `gzip on` может сжимать `.png`, `.woff2` — уже сжатые форматы. При gzip nginx пересчитывает Content-Length, но при определённых условиях (add_header, expires) расчёт может быть некорректен.

### 2. CDN/прокси перед Docker обрывает соединения
Cloudflare, хостинг-провайдер, или ISP может иметь лимит на размер ответа, таймаут, или проблему с buffering.

### 3. Docker контейнер — нехватка ресурсов
nginx worker исчерпал memory/file descriptors при одновременной загрузке множества ресурсов.

### 4. `add_header` + gzip конфликт
Директива `add_header` в nginx может конфликтовать с gzip-модулем: nginx добавляет хедеры, gzip меняет тело, но Content-Length уже зафиксирован.

## Рекомендуемые исправления

### Исправление 1: [КРИТИЧЕСКОЕ] nginx — отключить gzip для сжатых форматов

`frontend/Dockerfile` — добавить в inline nginx конфиг:
```nginx
# Добавить ПЕРЕД location блоками:
gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml;
gzip_min_length 1000;
# НЕ сжимать .png, .jpg, .woff2, .woff — они уже сжаты
```

И в location для статики:
```nginx
location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2)$ {
    expires 1y;
    add_header Cache-Control "public, immutable";
    gzip off;  # ← КРИТИЧЕСКИ: не сжимать статику повторно
}
```

### Исправление 2: [КРИТИЧЕСКОЕ] axios-интерцептор — обновить пути

`frontend/src/api/client.ts` — заменить:
```js
const onAuthPage = window.location.pathname === "/login" || window.location.pathname === "/register";
const publicPages = ["/", "/login", "/register"];
```
на:
```js
const onAuthPage = window.location.pathname.startsWith("/auth");
const publicPages = ["/", "/auth"];
```

Также заменить `window.location.href = "/login"` на `window.location.href = "/auth"`.

### Исправление 3: [ВАЖНОЕ] Workbox — включить иллюстрации в прекэш

`frontend/vite.config.ts` — обновить:
```js
globPatterns: ["**/*.{js,css,html,svg,woff2,png,webp,jpg}"],
globIgnores: [],
runtimeCaching: [
  // ... существующие правила для /api/ и /ws/
  {
    urlPattern: /\.(?:png|jpg|jpeg|svg|webp)$/,
    handler: "CacheFirst",
    options: {
      cacheName: "images",
      expiration: { maxEntries: 50, maxAgeSeconds: 30 * 24 * 60 * 60 },
    },
  },
],
```

Когда серверные проблемы (шаг 1) будут решены, прекэш иллюстраций обеспечит их доступность при последующих обновлениях SW.

### Исправление 4: [ЖЕЛАТЕЛЬНОЕ] Рассмотреть `registerType: "prompt"` вместо `"autoUpdate"`

При `autoUpdate` страница перезагружается автоматически при обновлении SW. На проде с нестабильным соединением это усугубляет проблемы. `prompt` даёт пользователю контроль.

## Порядок действий

1. **Сначала:** Шаг 1 (curl-диагностика сервера) → определить, в чём именно обрывается соединение
2. **Затем:** Исправление 1 (nginx gzip off для статики) → деплой → проверить
3. **Параллельно:** Исправление 2 (axios-интерцептор) → исправляет бесконечный рефреш
4. **После стабилизации:** Исправление 3 (Workbox конфиг) → прекэш иллюстраций
5. **Шаг 2** (проверка на другом устройстве) — для исключения клиентской проблемы
