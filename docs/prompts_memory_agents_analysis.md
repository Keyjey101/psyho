> Дата: 2026-05-04

## 1. Промпты Ники (orchestrator.txt)

### Проблема: "МОЖНО"-список сам порождает шаблоны

`orchestrator.txt:29-34` — список "разрешённых" фраз:
```
«Похоже, что...» / «Звучит как...» / «Мне кажется, что...»
```
Модель читает запрет на одни шаблоны, получает список одобренных — и ротирует именно их.  
**Исправление:** убрать конкретные примеры, заменить на принцип:
```diff
- «Похоже, что...» / «Звучит как...» / «Мне кажется, что...»
+ Конкретное наблюдение из слов пользователя, вопрос который открывает следующий слой,
+ метафора — всё это работает. Разнообразь форму от ответа к ответу.
```

### Проблема: жёсткое "каждый ответ начинается с..."

`orchestrator.txt:49` — формулировка "каждый" создаёт механику.  
**Исправление:**
```diff
- Каждый ответ начинается с признания того, что чувствует человек — не с анализа и не с советов.
+ Обычно начинай с признания состояния человека, но не всегда одним способом —
+ иногда это наблюдение, иногда вопрос, иногда тишина в одном предложении.
```

### Проблема: противоречие фаз и стилей

`orchestrator.py:81-92` — PHASE_INSTRUCTIONS для INTEGRATION жёстко предписывает "4-7-8 или 5-4-3-2-1".  
Но если стиль `direct` — пользователь не хочет навязанных техник; если `gentle` — "предложи упражнение" звучит директивно.  
Три слоя инструкций (базовый промпт + стиль + фаза) конфликтуют.  
**Исправление:** убрать конкретные техники из PHASE_INSTRUCTIONS, оставить "предложи практику под то, что возникло в сессии."

### Проблема: системный промпт раздувается

К моменту добавления memory + therapy_goals + style + address_form + gender + continuity + phase_instruction — system prompt превышает 1500 токенов. Модель начинает "забывать" ранние инструкции в пользу последних.  
**Исправление:** консолидировать секции, расставить явный приоритет важности.

## 2. Специализированные агенты

### Агенты не знают контекст вызова

`base.py` — `analyze()` получает только `user_message` и `history[-16:]`. Агентам неизвестны:
- текущая фаза сессии (INTAKE / WORK / CLOSE)
- preferred_style пользователя
- long_term_memory (включая упомянутую травму)
- что Ника уже пробовала в этой сессии

**Практический провал:** CBT-агент может рекомендовать домашнее задание на первом сообщении; соматический агент предложит интенсивную технику пользователю, у которого в памяти прописана травма.

### Только Jungian агент проверяет уместность

`jungian.txt` — единственный, у кого Шаг 2: "Уместен ли юнгианский анализ здесь?"  
CBT, IFS, Narrative, Somatic — все интерпретируют даже там, где их школа нерелевантна. CBT умеет находить когнитивные искажения в любой фразе.  
**Исправление:** добавить аналогичный "Шаг 0" во все агентные промпты.

### Формат вывода агентов мешает синтезу

Агенты возвращают структурированный текст с буллетами и заголовками (`**Автоматическая мысль**:`, `**Состояние нервной системы**:` и т.д.). Структура "просачивается" — Ника начинает давать ответы с нумерацией и перечислением.  
**Исправление:** попросить агентов писать выводы в виде абзацев-советов без заголовков.

### Длина анализа не масштабируется

`AGENT_PREAMBLE` (base.py) требует 300-400 слов. Пользователь написал "устал" — получает 600-800 слов суммарного анализа. Ника синтезирует это в 2-3 предложения. Огромный ratio потерь.  
**Исправление:** "соразмерно сообщению: 80-120 слов на короткое, до 300 на развёрнутое."

### Экспертный анализ добавляется в сообщение пользователя

`orchestrator.py:483-490`:
```python
user_content += f"""
Мнения экспертов:
{perspectives_text}
```
Анализ агентов — часть `user_content`, то есть Ника видит его как слова пользователя. Это структурно неверно.  
**Исправление:** передавать экспертный анализ отдельным `assistant`-сообщением или выделенным блоком в system prompt.

### Агенты используют glm-5 (основную модель)

`base.py:56` — `model=settings.ZAI_MODEL`. Два агента параллельно + оркестратор = три вызова glm-5 одновременно. Агентный анализ читает только Ника, не пользователь — это кандидат на `ZAI_SMALL_MODEL` для экономии скорости и стоимости.

## 3. Память о пользователе (memory_service.py)

### Главный пробел: запоминаются слова пользователя, но не наблюдения Ники

`memory_service.py:44`:
```python
user_msgs = [m["content"] for m in messages[-10:] if m["role"] == "user"]
```
Роль `assistant` фильтруется. Если Ника написала "Похоже, у вас есть паттерн избегания через работу" — этот инсайт теряется. Запоминаются только слова самого пользователя.  
**Исправление:** включить в контекст для извлечения и ответы Ники (помечая роли).

### 200 слов — слишком мало

За несколько сессий накапливается: имя, работа, семья, триггеры, паттерны, прогресс, цели. В 200 словах это не помещается — модель выбрасывает старое, чтобы добавить новое. Пользователь приходит в 5-ю сессию, а Ника "забыла" что он рассказывал в 1-й.  
**Исправление:** увеличить лимит до 400 слов, `max_tokens` до 700.

### Память обновляется после каждого сообщения

Дорого (один API-вызов на каждый обмен) и бессмысленно на первых 2-3 сообщениях.  
**Исправление:** обновлять каждые 3-4 обмена или при закрытии сессии.

### Память не доходит до агентов

Агенты анализируют без знания о пользователе. Если в памяти написано "травматический опыт в детстве" — соматический агент всё равно может предложить интенсивную технику.  
**Исправление:** передавать краткую выжимку памяти через параметр `focus` в `agent.analyze()`.

### Нет структуры — нет надёжности

Память — свободный текст. Предложение: добавить мягкую структуру, которая помещается в те же 300-400 слов, но надёжнее читается моделью:

```
[БИОГРАФИЯ] Имя: …, работа: …, семья: …
[ТЕМЫ] Тревога на работе, конфликты с матерью
[ЧУВСТВИТЕЛЬНО] Избегает разговора о детстве, болезненно реагирует на оценку
[ПРОГРЕСС] [тогда] паника перед встречами → [сейчас] применяет дыхание 4-7-8
```

Обновить `MEMORY_EXTRACT_PROMPT` в `memory_service.py` под этот формат.

## 4. Архитектурные наблюдения

### exchange_number=0 → WORK → агенты на первом сообщении

`orchestrator.py:398-404`:
```python
phase = (
    self._get_phase_adaptive(...)
    if exchange_number > 0 and max_exchanges > 0
    else SessionPhase.WORK
)
```
Первое сообщение всегда попадает в WORK и вызывает агентов. Они получают один абзац без контекста и генерируют полный анализ вхолостую.  
**Исправление:** первые 1-2 сообщения держать в INTAKE (agents не вызываются).

### Кэш тем не реагирует на резкую смену темы

`_TOPIC_CACHE_MSG_DRIFT = 6` — если пользователь переходит с тревоги на отношения, кэш держит старые темы ещё 6 сообщений. Вызываются CBT+somatic вместо нужных IFS+narrative.  
**Рассмотреть:** понизить drift до 3-4, или добавить детектор "тема сменилась резко".
