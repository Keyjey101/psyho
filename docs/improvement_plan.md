## 1. Технический аудит

### 1.1 Архитектура backend

**Общая оценка: 7/10** — Чистая модульная структура FastAPI с разумным разделением на routers/services/models. Однако есть архитектурные проблемы, которые ограничат масштабируемость.

#### Критические проблемы (P0)

| # | Проблема | Файл | Описание |
|---|---------|------|----------|
| T-01 | **SQLite в production** | `docker-compose.yml:24` | SQLite — файловая БД с единственным writer. При >10 одновременных пользователей WAL-режим начнёт деградировать. Миграция на PostgreSQL — обязательна для роста. |
| T-02 | **Orchestrator — singleton в модуле** | `routers/messages.py:25` | `orchestrator = Orchestrator()` создаётся один раз при импорте модуля. `_session_topic_cache` — in-memory dict без TTL и без очистки → утечка памяти при множестве сессий. |
| T-03 | **WebSocket auth без refresh** | `routers/messages.py:141` | JWT access token живёт 15 мин. Если WS-соединение открыто дольше, токен протухнет, но `decode_token()` не проверяет refresh. При реконнекте старый токен отклоняется. |

#### Высокий приоритет (P1)

| # | Проблема | Файл | Описание |
|---|---------|------|----------|
| T-04 | **`_generate_summary` может обрабатывать огромный текст** | `services/context.py:94` | Нет лимита на `formatted_messages` — при 40+ длинных сообщениях промпт может превысить контекстное окно модели. |
| T-05 | **Background tasks без error boundary** | `routers/messages.py:355-371` | `asyncio.create_task()` без обёртки в try/except. Если task выбросит исключение, оно будет потеряно. |
| T-06 | **`_memory_counters` — in-memory dict** | `routers/messages.py:40` | При перезапуске сервера счётчики теряются. При multiple workers — не шарятся. |
| T-07 | **`init_db()` выполняет ALTER TABLE** | `database.py:27-30` | Это должно быть в Alembic миграции. `ALTER TABLE` в runtime — хак, который маскирует проблемы миграций. |
| T-08 | **Rate limiter не применяется к WS** | `main.py:19` | Slowapi настроен, но `@limiter.limit()` не используется ни на одном эндпоинте. Rate limiting декларативен, но не подключён. |
| T-09 | **Telegram OTP хранит code в plaintext** | `models/models.py:97` | `TelegramVerificationCode.code` — plain string, в отличие от `EmailVerificationCode.code_hash`. При утечке БД — утечка OTP-кодов. |
| T-10 | **`datetime.utcnow()` в `auth.py:335`** | `routers/auth.py:335` | Используется `datetime.utcnow()` (naive) вместо `datetime.now(timezone.utc)` (aware). Несогласованность с остальным кодом, может привести к багам при сравнении. |

#### Средний приоритет (P2)

| # | Проблема | Файл | Описание |
|---|---------|------|----------|
| T-11 | **`_sanitize_memory` — простой substring match** | `agents/orchestrator.py:117-122` | Memory injection patterns легко обойти (кириллица, юникод, URL-encoding). Нужен более robust подход или полное отсутствие пользовательского ввода в system prompt. |
| T-12 | **Crisis detector — keyword matching** | `agents/orchestrator.py:101-106` | Простой substring search без контекста → ложные срабатывания («устал жить в этом городе» = «не хочу жить»). Нужен LLM-based или хотя бы regex с отрицаниями. |
| T-13 | **`_select_agents` ограничивает до 2 агентов** | `agents/orchestrator.py:199-214` | Двойной break при ≥2 agents, но внутренний цикл не проверяет все topics. Может выбрать 2 агента от одной темы, проигнорировав вторую. |
| T-14 | **Schemas дублируют ручную сериализацию** | `routers/personality.py`, `routers/mood.py` | Многие эндпоинты возвращают `dict` вместо Pydantic response_model. Несогласованно с `sessions.py`, где схемы используются. |
| T-15 | **`get_current_user` делает DB-запрос на каждый запрос** | `middleware/auth.py:42` | Нет кеширования. При активном использовании — лишняя нагрузка на БД. Можно кешировать на время жизни access token (15 мин). |
| T-16 | **No pagination in `list_sessions`** | `routers/sessions.py:19-26` | Все сессии грузятся без лимита. При >100 сессий у пользователя — медленно. |
| T-17 | **`_extract_and_save_task` создаёт максимум 1 задачу на сессию** | `routers/messages.py:85` | Проверка `existing.scalar() == 0` — если задача уже есть, новые упражнения не сохранятся как задачи. |

### 1.2 Архитектура frontend

**Общая оценка: 7.5/10** — Современный стек (React 19, TypeScript, Zustand, React Query), хорошие паттерны. Но есть проблемы с типобезопасностью и производительностью.

#### Высокий приоритет (P1)

| # | Проблема | Файл | Описание |
|---|---------|------|----------|
| T-18 | **Type casting через `as any`** | `pages/Chat.tsx:48,65` | `(user as any).name`, `(user as any)?.profile?.memory_enabled` — тип `User` из `types/index.ts` уже включает `profile`. Касты маскируют несоответствие типов. |
| T-19 | **`useChat` не чистит ref при смене сессии** | `hooks/useChat.ts` | `streamingContentRef.current` не обнуляется при `sessionId` change в некоторых сценариях. |
| T-20 | **Refresh token race condition** | `api/client.ts:31-51` | Если два параллельных запроса получают 401, оба пытаются refresh — возможен двойной refresh или использование устаревшего токена. Нужен mutex. |
| T-21 | **`window.location.href = "/auth"` в interceptor** | `api/client.ts:49` | Полная перезагрузка страницы вместо `navigate()` — теряется состояние, не используется React Router. |

#### Средний приоритет (P2)

| # | Проблема | Файл | Описание |
|---|---------|------|----------|
| T-22 | **Inline styles вместо Tailwind** | `components/landing/Hero.tsx:48-56`, `pages/PersonalityPage.tsx:89` | Inline `style={{}}` для gradient и SVG вместо Tailwind-классов. Несогласованно. |
| T-23 | **`MoodPage` грузит все данные без React Query** | `pages/MoodPage.tsx:69-81` | Ручные `api.get()` + `useState` + `useEffect` вместо React Query → нет кеширования, нет refetch, нет loading/error states. |
| T-24 | **`PersonalityPage` — то же самое** | `pages/PersonalityPage.tsx:209-215` | Ручной data fetching вместо React Query. Дублирование паттерна. |
| T-25 | **`OnboardingFlow` не проверяет `step === TOTAL_STEPS`** | `pages/OnboardingFlow.tsx:117-123` | `handleSkip` для step 5 вызывает `handleFinish()`, но кнопка «Пропустить» не появляется для step 5. Логика запутана. |
| T-26 | **WS_agents_used interface naming** | `types/index.ts:61` | `WS_agents_used` — не PascalCase. Должно быть `WSAgentsUsedMessage`. |

### 1.3 Безопасность

| # | Проблема | Серьёзность | Описание |
|---|---------|-------------|----------|
| S-01 | **`secure=True` на cookies без HTTPS fallback** | Средняя | `auth.py:64` — `secure=True` означает, что cookies не отправляются по HTTP. В development (без HTTPS) авторизация не работает через cookies. Нужно условие: `secure=settings.ENVIRONMENT == "production"`. |
| S-02 | **`TEST_PASSWORD_CODE` — мастер-пароль** | Высокая | В production, если `TEST_PASSWORD_CODE` установлен в `.env`, любой может залогиниться под любой email. Это backdoor. |
| S-03 | **OTP plaintext в Telegram** | Средняя | `TelegramVerificationCode.code` — plaintext. См. T-09. |
| S-04 | **Нет CSRF защиты** | Средняя | Cookies с `samesite="lax"` — частичная защита, но `PATCH`/`POST` запросы от поддоменов могут быть CSRF. |
| S-05 | **Логирование OTP-кода** | Высокая | `auth.py:146` — `logger.info("verify-code attempt", ..., code=body.code, test_code=settings.TEST_PASSWORD_CODE)` — OTP и master-password пишутся в лог. |

### 1.4 Принципы проектирования

| Принцип | Оценка | Комментарий |
|---------|--------|-------------|
| **SOLID** | 6/10 | SRP нарушен в `messages.py` (WebSocket + бизнес-логика + background tasks в одном файле). OCP — ок (новые агенты добавляются легко). DIP — частично (прямые импорты client вместо интерфейса). |
| **DRY** | 5/10 | Сериализация в dict повторяется в `personality.py`, `mood.py`, `tasks.py`. Создание пользователя дублируется 5 раз в `auth.py` (send-code, verify-code, tg/check, tg/mini-app, telegram). |
| **KISS** | 8/10 | Код в целом простой и понятный. `messages.py` — исключение (381 строка, много ответственностей). |

## 2. Продуктовый анализ

### 2.1 Уникальное ценностное предложение (UVP)

**«Ника — ИИ-психолог с мультиагентной системой: 6 терапевтических подходов в одном чате»**

Сильные стороны UVP:
- Мультиагентность — реальное техническое преимущество перед ChatGPT/single-model
- Кризисный детектор — этическое преимущество
- Персонализация (ты/вы, стиль, гендер, память) — глубокая настройка
- Психопортрет — gamification + визуализация прогресса
- Telegram mini-app — низкий барьер входа

Слабые стороны UVP:
- Нет дифференциации от «просто ChatGPT с терапевтическим промптом» для нетехнических пользователей
- Нет социального доказательства (отзывы, кейсы)
- Нет бесплатного демо без регистрации

### 2.2 Целевая аудитория

| Сегмент | Размер | Потребность | Текущее покрытие |
|---------|--------|------------|-----------------|
| **Молодые взрослые 18-35** с тревогой/стрессом | Большой | Доступная психологическая поддержка 24/7 | Хорошо |
| **Люди, не готовые к живому терапевту** | Средний | Анонимность, отсутствие обязательств | Хорошо |

| **Профессионалы (психологи)** | Малый | Инструмент дополнения практики | Не покрыто |

## 3. UI/UX экспертиза

### 3.1 Визуальная система

**Цветовая палитра:** Тёплая terracotta (`#B8785A`) как primary, cream/beige фоны (`#FAF6F1`). Согласованная и уютная — подходит для therapeutic контекста.

**Сильные стороны:**
- Единая цветовая система через CSS custom properties
- Dark mode реализован последовательно
- WebP-оптимизированные иллюстрации с PNG fallback
- `font-serif` для заголовков — создаёт теплоту

**Проблемы:**

| # | Проблема | Где | Рекомендация |
|---|---------|------|-------------|
| UX-01 | **Типографическая иерархия плоская** | Везде | `text-[15px]` используется для body, подзаголовков и кнопок. Нужна чёткая шкала: `text-xs` → `text-sm` → `text-base` → `text-lg` → `text-xl`. |
| UX-02 | **`font-serif` без указания шрифта** | `Hero.tsx:73`, `Landing.tsx:54` | `font-serif` зависит от browser default (Georgia/Times). Нужно подключить явный веб-шрифт (Merriweather, Playfair Display) для единообразия. |
| UX-03 | **Иконки太小 в Sidebar** | `Sidebar.tsx:75` | 8×8 avatar — неразличим на мобильных. Увеличить до 10×10 или 12×12. |
| UX-04 | **Нет skeleton/loading states** | `MoodPage`, `PersonalityPage` | Белый экран при загрузке → spinner. Нужны skeleton-каркасы. |
| UX-05 | **Landing не использует `Features` компонент** | `Landing.tsx:79-84` | `Features.tsx` импортирован в Landing, но на лендинге он не рендерится. Бесполезный компонент или пропущенный section. |
| UX-06 | **Progress bar не информативен** | `SessionProgress.tsx` | Тонкая полоска 5px — незаметна. Нужен более заметный индикатор с текстом фазы (Intake/Work/Integration). |

### 3.2 User Journey анализ

#### Journey 1: Новый пользователь через сайт
```
Лендинг → «Начать разговор» → /auth (Telegram) → /auth/email → /auth/verify → /onboarding → /chat
```

**Точки трения:**
1. **3 шага до чата** (auth → verify → onboarding) — потеря ~40% пользователей
2. **Нет предварительного демо** — пользователь не понимает ценность до регистрации
3. **Onboarding — 5 шагов** — слишком много для первого опыта

**Рекомендации:**
- Сократить onboarding до 2-3 шагов (имя + цели)
- Показать демо-ответ Ники на лендинге
- Опциональный onboarding с возможностью «Начать сейчас»

#### Journey 2: Telegram Mini App
```
Бот → «Открыть приложение» → /chat (auto-auth) → /onboarding → /chat
```

**Точки трения:**
1. **Telegram Android bug** (`initData` пустой) — fallback реализован, но опыт может быть нестабильным
2. **После onboarding — пустой чат** без подсказки «Привет! Напиши мне что-нибудь»

#### Journey 3: Возврат пользователя
```
Открыть приложение → /chat → Список сессий → Выбрать сессию
```

**Точки трения:**
1. **Нет push-уведомлений** о невыполненных заданиях
2. **Нет «Привет! Как ты?» при возвращении** — missed engagement opportunity
3. **Sidebar загружает все сессии** — при >20 сессий трудно найти нужную

### 3.3 Точки вовлечения (Engagement Points)

| Точка | Текущее состояние | Рекомендация |
|-------|-------------------|-------------|
| Первый вход | Onboarding (5 шагов) | Сократить до 2-3, добавить демо-сообщение от Ники |
| Во время сессии | Agent badges, progress bar | Добавить «фазу сессии» визуально, micro-feedback |
| Конец сессии | SessionEndCard (mood + task) | Хорошо! Добавить «share achievement» |
| Между сессиями | Ничего | Push-уведомления, e-mail дайджест, reminder о задачах |
| Психопортрет | После 3 сессий | Хорошо! Но нет оповещения «Психопортрет обновлён!» |

## 4. Брейншторминг

### 4.1 Скрытые проблемы

| # | Проблема | Почему скрыта | Влияние |
|---|---------|--------------|--------|
| H-01 | **Cost per conversation** | Не отслеживается token usage | При росте users — неконтролируемые расходы на AI API |
| H-02 | **Hallucination agents_used** | Агенты вызываются, но их «мнения» могут не попасть в финальный ответ | Пользователь видит badge «КПТ», но ответ может не содержать КПТ-инсайтов |
| H-03 | **Memory poisoning** | `_sanitize_memory` прост | Через многократные сообщения можно «загрязнить» long_term_memory |
| H-04 | **No data export/GDPR** | Нет self-service удаления данных | Юридический риск в регионах с GDPR/ФЗ-152 |
| H-05 | **Single point of failure: ZAI API** | Нет fallback провайдера | Если ZAI недоступен — всё приложение не работает |
| H-06 | **Session continuation — partial context** | `continue_session` загружает только 20 последних сообщений | Длинные сессии теряют ранний контекст |
| H-07 | **No A/B testing framework** | Монолитный промпт | Невозможно тестировать разные варианты ответов |
